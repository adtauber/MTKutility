Download Log

The log download function transfers track point data from your GPS logger to a binary file on your phone.  Click the Download Log button to start the process.  Download progress is shown by the progress circle turning green and the tally of bytes transfered increasing.  The file is created in your phone's download folder in the sub-folder MTKutility/bin.  You can transfer the binary file to another location using your phone's File Manager app.

The file created is a raw dump of your GPS logger memory.  It cannot be used directly by track display apps.  You can create a GPX file with MTKutility.

I am not aware of an Android app that will create a KML file.  The PC application I know of for KML file creation is BT747.  It will create a KML file from the MTKutility binary.

I'll put a plug in for the GPX Viewer PRO app as a good Android app for viewing GPX files.  It opened a 17 MB file very quickly and can be used offline if you have the map downloaded.

dealing with problems that arise
The log download function seems to be the most problematic for some device combinations. The problem manifests as a loss of Bluetooth connection for no discernible reason.  MTKutility now has auto-resume built in.  The app will restart and resume the download at the point of failure.  However, the auto-resume will fail if a Bluetooth connection cannot be established on resumption.  You will see a red message when this happens and you need to manually attempt restart by clicking the Download Log button.

I verified that a download with multiple resumes is complete with an MD5 comparison of this GPX file to the GPX from an uninterrupted download of the same data.  You can abort resuming and reset the start point to 0 with the reset button while the download is in progress.

Leaving the download screen while the download is in progress is disabled.  You will not be able to leave MTKutility if the update hangs without resuming. You will have to kill the app from your android system.  Make sure you run MTKutility right after you kill it to reset your logger.

The log download has four preferences:
  • download send/receive delay
  • command retry delay increment
  • Download Block Size
  • disable NMEA for download/AGPS update
The  download send/receive delay setting is the milliseconds wait time between the request for data and the read attempt.  Decreasing this will speed up the download but it may also result in download failures.

command retry delay increment is there to assist with resolving download issues.  Leave this at 0 as the only way to understand the effect is to examine a verbose level 2 log.  Please refer to the app preferences help for the detail on this and the Download Block Size setting.

The disable NMEA for download/AGPS update default is off to accommodate GPS devices that function erratically when NMEA sentence output is turned off.  However many loggers support full suppression of  NMEA output and will process the download and update faster when there isn't NMEA traffic to the Android device.

my test results
To test this I maxed the capacity of my Qstarz BT-Q1000X at 225174 records (4,194,290 bytes).  On my ASUS ZenFone 4 with Android 8.1.0 it took 27 minutes 44 seconds to download with the download send/receive delay set at 50 and disable NMEA for download/AGPS update set.

This took 34 minutes 8 seconds on my ancient Android 4.1.2 Samsung SGH-I717D Note with the download send/receive delay set at 30 and disable NMEA for download/AGPS update set.  I consider this a real success as the download would not work on this phone in my Feb 2019 rewrite.
